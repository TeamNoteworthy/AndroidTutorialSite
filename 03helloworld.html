Adjust Manifest

Let's look at the application's Manifest file. On the Project panel, expand the manifests folder and select AndroidManifest.xml.

Figure 3-7 shows the Manifest file for the entire application. The first line defines that the file uses an XML layout. The second line defines the namespace with xmlns:android

(similar to the C++ concept of namespaces--it prevents variable name collisions). The value of this attribute points to some line of text that must differentiate the elements inside of it from other elements. It doesn't have to be a real website--it doesn't have to be a website at all, but it typically points to some resource that explains something about the project. The next line defines the package, which must be unique for all applications on a device, and therefore Google Play requires unique package names across all those applications submitted.

A quick tutorial on XML (eXtendable Markup Language).

It is a language that describes data and it consists of tags contained in angle brackets similar to HTML. For each attribute, there is an identifier to mark the kind of attribute. Before the first closing angle bracket, there can be an arbitrary number of attributes listed with the syntax <tagName attribute0="value0" attribute1="value1">. Namespaces prevent collisions of similarly named variables across a large number of XML files that may be used for a similar purpose. To access a namespace, use this syntax: <tagName namespace:attribute="value">. All the variables you'll edit are in the android: namespace as defined in the second line. Tags (once the initial angle bracket has been closed) can be nested so that text or other attributes apply in a nested-tree fashion to the tags which contain them. Learn more about XMLhere.



The application tag defines the app's icon, its name, and its activities, among other things. At least one Activity must have an Intent designating it as the LAUNCHER Activity so Android can put that Activity on the Home screen (you could potentially have multiple launching activities). Making the intent MAIN indicates that the intent needs no further information in an intent other than that the Activity should launch (as opposed to being prompted by another Activity). Read more here.

To jazz up our app a little, we're going to add a theme: @android:style/Theme.Holo.Light.DarkActionBar. So within the Activity tag (the particular order of attributes isn't important, but we'll insert it in a line after the android:label attribute), add this: 


    android:theme="@android:style/Theme.Holo.Light.DarkActionBar"

with the tag closing afterwards.

The @ symbol means that we're referring to a resource--if it is followed by android:, it's defined in the Android system's general resources. Otherwise, it refers to a resource made in the particular project.

So now the manifest should look like this: 


<?xml version="1.0" encoding="utf-8"?>
<manifest xmlns:android="http://schemas.android.com/apk/res/android"
package="edu.gatech.androidnoteapp" >
        <application
        android:allowBackup="true"
        android:icon="@drawable/ic_launcher"
        android:label="@string/app_name"
        android:theme="@style/AppTheme" >
        <activity
            android:name=".NoteListActivity"
            android:label="@string/app_name"
            android:theme="@android:style/Theme.Holo.Light.DarkActionBar">
            <intent-filter>
                <action android:name="android.intent.action.MAIN" />
                <category android:name="android.intent.category.LAUNCHER" />
            </intent-filter>
        </activity>
    </application>
</manifest>

Adjust Layout

Now navigate back to the code that's already open in a tab: activity_note_list.xml.

Figure 3-9 shows the completed manifest, which dictates how the screen will be laid out. Android separates the layout from the code in order to avoid the clunky GUI code that is found in Java Swing (and to make debugging easier). Thus, you can use the drag-and-drop editor (the Design tab on the bottom of the pane) to modify the layout, but for now we'll show you the manual way. However, for the sake of being able to edit layout at runtime, Android does allow for adding and modifying layout in the code.

Notice the Design and Text tabs at the bottom of the edit pane, which allow you to switch between the two ways of editing the same file.

In the Text editor, at the top you'll see a few lines describing how the layout is to be padded. Right now, we're going to modify the layout_width and layout_height attributes of the TextView tag slightly. Change the value of those attributes from wrap_content to match_parent. While wrap_content tells that UI object to take up as much space as its content needs, match_parent tell the UI object to take up as much space as the parent will allow. With a little adjusting whitespace (which XML ignores), the resulting XML file should look like this. 


<RelativeLayout xmlns:android="http://schemas.android.com/apk/res/android"
    xmlns:tools="http://schemas.android.com/tools" android:layout_width="match_parent"
    android:layout_height="match_parent" android:paddingLeft="@dimen/activity_horizontal_margin"
    android:paddingRight="@dimen/activity_horizontal_margin"
    android:paddingTop="@dimen/activity_vertical_margin"
    android:paddingBottom="@dimen/activity_vertical_margin" tools:context=".NoteListActivity">
    <TextView android:text="@string/hello_world"
        android:layout_width="match_parent"
        android:layout_height="match_parent" />
</RelativeLayout>

Read more about layout here.

NoteListActivity

NoteListActivity is found in the Java folder. There's a series of folders under it for each sub-domain in your company's domain--that's just a Java convention.

The class imports a few necessary Android clases, then declares the class and extends ActionBarActivity. The base class all Activities need to extend is (*drum roll*) Activity, but there are various specialized Activities for various Activity templates. Since the Theme we are using is not supported by ActionBarActivity, we're going to downgrade to just Activity. When that results in an error from not having the class imported, click on the word Activity and press ALT + Enter. A dialog will pop up asking you how to automatically fix the problem. The first way, by importing the class, is what we want, so just press Enter and it will be done. You can also remove the import line for android.support.v7.app.ActionBarActivity.

From Android's website, Figure 3-10 shows an ActionBar.

Nothing too exciting, but we'll use it later on.

The @Override line is a directive to the compiler to let it know that the next line will override a method previously defined by one of the parent classes.

The onCreate() method responds to what happens when the app first appears. The line 

super.onCreate(savedInstanceState);

is merely Android's way of restoring some metadata about the state of the GUI from the last time the app opened. For now, we don't need to know its details.

setContentView(R.layout.activity_note_list) tells the compiler to define how the app looks based on the XML layout as defined by activity_note_list.xml (the resource within the layout folder).

The createOptionMenu() method determines how the menu that comes forth from the button when it is pressed. MenuInflater is an XML parser that parses the XML file found at R.menu.menu_note_list.xml. We'll go over that in more detail when we edit the Options menu.

Finally, onOptionsItemSelected() determines how to react when the user selects one of the options in the Option menu. It usually involves a large switch statement to determine which menu item has been pressed, and then reacts accordingly. Right now, it takes no action--it merely returns true (meaning no errors were incurred) without taking any action.

The finished file should look like this: 


package edu.gatech.androidnoteapp;
import android.app.Activity;
import android.os.Bundle;
import android.view.Menu;
import android.view.MenuItem;

public class NoteListActivity extends Activity {
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        setContentView(R.layout.activity_note_list);
    }
    
    @Override
    public boolean onCreateOptionsMenu(Menu menu) {
        // Inflate the menu; this adds items to the action bar if it is present.
        getMenuInflater().inflate(R.menu.menu_note_list, menu);
        return true;
    }
    @Override
    public boolean onOptionsItemSelected(MenuItem item) {
        // Handle action bar item clicks here. The action bar will
        // automatically handle clicks on the Home/Up button, so long
        // as you specify a parent activity in AndroidManifest.xml.
        int id = item.getItemId();
        //noinspection SimplifiableIfStatement
        if (id == R.id.action_settings) {
            return true;
        }
        return super.onOptionsItemSelected(item);
    }
}
